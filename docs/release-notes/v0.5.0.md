# Spec Lifecycle Manager v0.5.0

## Highlights

- Adds `slm`, a short public command for inspecting active specs, task state,
  requirements, priorities, next work, and closed-spec history.
- Keeps interactive and automated use aligned through concise plain-text views
  and deterministic versioned JSON produced from the same normalized records.
- Ships and tests the command through the npm tarball while retaining `slc` as
  a separate checkout-only maintainer tool and MCP as the preferred agent
  interface.

## Added

- Added `slm specs`, `slm tasks`, `slm next`, `slm requirements`, and
  `slm history` with deterministic active-spec selection and actionable
  ambiguity errors.
- Added task filters for literal pending, complete, all open states, repeatable
  normalized states, and dependency-aware next-task selection.
- Added requirement priority and missing-priority filters, plus archive-backed
  history filters for retained and removed specs.
- Added `--json`, `-C PATH`, and `--repo PATH` support across the inspection
  commands.
- Added a repo-root `./slm` launcher for source-backed testing without changing
  a user-wide installation.

## Changed

- `slm` is now the sole npm executable. The unused
  `spec-lifecycle-manager` and `ai-spec-lifecycle` executable aliases have been
  removed; existing scripts must call `slm`.
- Package installation now uses `slm install`. The dispatcher keeps installer
  routing in-process and launches inspection commands through the packaged
  Python resolver without a shell.
- Bare `slm` is equivalent to `slm specs`. Commands that take a spec accept a
  full ID, unique numeric prefix, slug, or package path, and select implicitly
  only when exactly one active spec exists.
- The public CLI reuses shared lifecycle parsing, state, priority, next-task,
  resolution, health, and archive contracts rather than defining CLI-only
  semantics.

## Fixed

- Avoided repeated full package lint while building the active-spec inventory
  by reusing scan health and shared task projections.
- Updated closed-spec history destinations that still named the replaced
  `npm-install.js` entrypoint, restoring a clean archive-index contract.
- Kept table and JSON selection, ordering, and state meanings synchronized by
  rendering both from the same normalized records.

## Documentation

- Added the public CLI path, examples, filters, state semantics, JSON contract,
  root selection, error behavior, and read-only boundary to the README and
  runtime reference.
- Documented the distinct responsibilities of public `slm`, checkout-only
  `slc`, and structured MCP tools.
- Updated install and packaging guidance for the sole-bin migration,
  interpreter resolution, isolated tarball validation, and source-backed
  development.

## Validation

- The final `npm run validate` release-candidate run passed with 353 Python
  tests and 32 Node tests, plus lifecycle scan, archive-index and prompt
  validation, package contract, source/bundle sync guard, npm package dry-run,
  and whitespace checks.
- The checkout-only maintainer CLI suite passed 17 tests.
- An isolated installed-tarball smoke verifies the `slm` shim, absence of the
  legacy shims, `slm --help`, `slm specs --json`, and `slm install --help`.
- The cross-platform workflow runs the packaged smoke on Windows, macOS, and
  Linux.

## Known Issues

- Existing scripts that invoke either removed executable alias must migrate to
  `slm`.
- Existing Codex and Claude sessions must be restarted or reloaded after the
  new package is installed.
- The repo-root `./slm` launcher is a source-development convenience. On
  Windows, invoke the same file as `node slm`; installed packages provide the
  normal platform-specific npm shim.
