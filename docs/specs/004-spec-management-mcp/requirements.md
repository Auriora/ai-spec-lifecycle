# Requirements

## Introduction

This spec defines requirements for a repository-local MCP server and related
workflow surfaces that support spec lifecycle management. The server should help
agents and developer operators create, inspect, lint, reconcile, review,
promote, and close implementation spec packages without replacing the existing
`spec-lifecycle-manager` skill.

The intended architecture is layered:

- Skills encode stable agent behavior and repository workflow judgment.
- MCP resources expose current spec, template, governance, and health context.
- MCP tools perform deterministic checks and structured operations.
- MCP prompts provide user-invoked workflow entry points and parameterized
  review packets.
- Hooks run narrow checks at file-change, task-completion, and closure gates.
- Cheap or fast agents handle bounded semantic review packets generated by the
  MCP surface.

- Durable MCP integration guidance should be promoted to `docs/design/` or

  should be updated.

## Goals

- Define whether MCP prompts, Skills, or both should drive spec-management
  workflows.
- Specify an MCP capability set for active spec discovery, linting,
  reconciliation, promotion planning, closure checks, and review-packet
  generation.
- Define deterministic document linters for spec artifact types.
- Define hook integration points for spec lifecycle quality gates.
- Define safe boundaries for using fast or cheap agents to offload bounded
  review and analysis work.

## Non-Goals

- Replace the `spec-lifecycle-manager` skill.
- Build a fully autonomous implementation manager that can close specs without
  operator review.
- Force every repository to adopt this repository's fallback templates.
- Implement Spec Kit-specific extension behavior unless it maps cleanly to this
  repository's lifecycle model.
- Require MCP prompt support from clients that do not expose MCP prompts well.

## Glossary

| Term | Meaning |
| --- | --- |
| Skill | Portable instruction package such as `SKILL.md` plus references that teaches an agent a domain workflow. |
| MCP prompt | User-invoked prompt template exposed by an MCP server through `prompts/list` and `prompts/get`. |
| MCP resource | Read-oriented MCP surface that exposes structured context such as active specs, templates, indexes, or health summaries. |
| MCP tool | Schema-defined callable operation that can perform deterministic checks, parse files, generate reports, or propose changes. |
| Hook | Local event-triggered check, such as a pre-commit lint or post-task evidence check. |
| Review packet | Bounded prompt plus context bundle suitable for a cheap or fast agent review pass. |
| Spec artifact | One document in a spec package, such as `requirements.md`, `design.md`, `tasks.md`, `verification.md`, or `change-impact.md`. |
| Traceability matrix | Bidirectional mapping between requirements, acceptance criteria, design sections, task IDs, verification evidence, durable targets, and open decisions. |

## Requirements

### Requirement 1: Preserve Skill Authority

**User Story:** As a developer operator, I want the existing skill to remain
the authority for lifecycle judgment, so that a new MCP server improves
execution without fragmenting the workflow.

#### Acceptance Criteria

1. GIVEN a spec workflow request, WHEN the agent has the
   `spec-lifecycle-manager` skill available, THEN the workflow SHALL treat the
   skill as the authoritative procedural guide.
2. GIVEN an MCP tool result, WHEN the result conflicts with skill guidance,
   THEN the agent SHALL surface the conflict and require reconciliation before
   proceeding.
3. IF repository governance or templates conflict with MCP-generated guidance,
   THEN repository governance and templates SHALL win.

### Requirement 2: Use MCP Prompts For Explicit Workflow Entry Points

**User Story:** As a user, I want reusable MCP prompts for common spec
workflows, so that I can start structured work without remembering every
lifecycle instruction.

#### Acceptance Criteria

1. GIVEN a client that supports MCP prompts, WHEN the user lists available
   prompts, THEN spec workflow prompts are discoverable with clear names,
   descriptions, and arguments.
2. GIVEN a prompt invocation, WHEN required arguments are missing, THEN the
   prompt SHALL request or validate those arguments before generating workflow
   instructions.
3. WHERE prompt support is unavailable or weak in a client, THE SYSTEM SHALL
   still support the same workflow through Skills and MCP tools/resources.
4. IF a prompt includes server-managed context, THEN it SHALL reference
   resources explicitly and avoid hiding policy decisions inside generated
   prose.

### Requirement 3: Provide Spec Resources

**User Story:** As an agent, I want structured read access to specs,
templates, governance, and status, so that I can reason over the lifecycle
without ad hoc file discovery.

#### Acceptance Criteria

1. GIVEN a repository root, WHEN the agent reads active spec resources, THEN
   it receives active package paths, artifact inventory, status, and detected
   format.
2. GIVEN a spec ID, WHEN the agent reads its summary resource, THEN the result
   includes requirements, tasks, open decisions, verification state, and
   durable-source references.
3. GIVEN template resources, WHEN the agent requests a template by artifact
   type, THEN the resource returns the effective repository template or records
   that the skill fallback template is being used.
4. IF old-format artifacts are present, THEN resources SHALL identify them
   without automatically migrating them.

### Requirement 4: Provide Deterministic Spec Linters

**User Story:** As a maintainer, I want deterministic lint checks for each
document type, so that template drift and missing evidence are caught before
agents claim completion.

#### Acceptance Criteria

1. GIVEN a `requirements.md`, WHEN it is linted, THEN the linter checks
   frontmatter, durable source baseline, goals, non-goals, user stories, EARS
   acceptance criteria, correctness properties, and success criteria.
2. GIVEN a `design.md`, WHEN it is linted, THEN the linter checks high-level
   design, low-level design, data flow or interfaces where applicable,
   operational considerations, and open questions.
3. GIVEN a `tasks.md`, WHEN it is linted, THEN the linter checks task IDs,
   dependency references, checkbox state, affected files, acceptance criteria,
   and evidence for completed tasks.
4. GIVEN optional artifacts, WHEN they are linted, THEN the linter applies
   artifact-specific checks for change impact, verification, research,
   quickstart, and open decisions.
5. IF a document intentionally deviates from a template, THEN the linter SHALL
   support an explicit waiver with reason and residual risk.

### Requirement 5: Reconcile Specs Against Durable Docs And Evidence

**User Story:** As a developer operator, I want structured reconciliation
reports, so that resumed specs reveal drift before implementation continues.

#### Acceptance Criteria

1. GIVEN a spec package, WHEN reconciliation runs, THEN it classifies detected
   drift as `spec stale`, `code incomplete`, `durable docs stale`,
   `decision unresolved`, `implemented but unverified`, `intentionally
   deferred`, or `governance conflict`.
2. GIVEN completed task checkboxes, WHEN evidence is missing, THEN
   reconciliation SHALL flag the task as incomplete or unverified.
3. GIVEN durable-source references, WHEN referenced docs are missing or stale,
   THEN reconciliation SHALL identify the missing or stale target.
4. IF reconciliation cannot inspect required code or docs, THEN the report
   SHALL record the blind spot instead of claiming a pass.

### Requirement 6: Select The Next Safe Task Slice

**User Story:** As an implementation agent, I want the MCP server to identify
the next runnable task slice, so that work follows dependencies and evidence
rules.

#### Acceptance Criteria

1. GIVEN a task graph, WHEN next-task selection runs, THEN it returns tasks
   whose dependencies are complete and verified.
2. GIVEN parallel-safe tasks, WHEN the selector marks `[P]` tasks, THEN it
   SHALL include write-scope and dependency reasons.
3. GIVEN blocked tasks, WHEN the selector reports them, THEN it SHALL include
   the blocking task, decision, evidence, or validation gap.
4. IF no safe task exists, THEN the selector SHALL return a concrete blocker
   rather than a generic failure.
5. GIVEN a selected task ID, WHEN next-task selection returns task context,
   THEN it SHALL include related requirements, acceptance criteria, design
   sections, change-impact entries, verification expectations, durable targets,
   and open decisions.
6. IF a traceability matrix is missing or stale, THEN the selector SHALL infer
   task context from the full spec package and report the matrix gap.

### Requirement 6A: Provide Traceability Lookup

**User Story:** As an implementation agent, I want to look up a task ID and
receive the relevant specification context, so that I do not implement from task
wording alone.

#### Acceptance Criteria

1. GIVEN a spec ID and task ID, WHEN traceability lookup runs, THEN it returns
   the task row, related requirements, acceptance criteria, design sections,
   change impact, verification expectations, durable targets, and open
   decisions.
2. GIVEN a requirement ID, WHEN reverse lookup runs, THEN it returns related
   design sections, tasks, verification entries, and durable targets.
3. GIVEN a design section, WHEN reverse lookup runs, THEN it returns related
   requirements, tasks, files or interfaces, and verification entries.
4. IF any referenced artifact is missing, stale, or ambiguous, THEN lookup
   SHALL report the gap instead of inventing context.

### Requirement 7: Generate Review Packets For Cheap Agents

**User Story:** As a developer operator, I want bounded review packets for
fast agents, so that parallelism improves confidence without creating noisy or
overlapping work.

#### Acceptance Criteria

1. GIVEN a review type, WHEN a packet is generated, THEN it includes a bounded
   question, input artifacts, expected output schema, constraints, and stop
   conditions.
2. GIVEN multiple review packets, WHEN they are generated for parallel agents,
   THEN their scopes SHALL be read-only unless the operator explicitly assigns
   a write scope.
3. IF review outputs conflict, THEN the lead agent SHALL reconcile the conflict
   before implementation proceeds.
4. WHERE cheap-agent review is used, THE SYSTEM SHALL record which packet ran,
   which model or agent class was used when known, and how findings were
   accepted, rejected, or deferred.

### Requirement 8: Support Hooks Without Over-Automating

**User Story:** As a maintainer, I want hooks for narrow spec lifecycle checks,
so that common mistakes are caught locally without turning the workflow into a
heavy gate for every small edit.

#### Acceptance Criteria

1. GIVEN changed spec files, WHEN a spec-lint hook runs, THEN it lints only
   affected packages and reports actionable findings.
2. GIVEN a task marked complete, WHEN a completion hook runs, THEN it checks
   for evidence before allowing or warning on completion.
3. GIVEN a closure attempt, WHEN a closure hook runs, THEN it checks durable
   promotion, verification evidence, unresolved decisions, and active-index
   status.
4. IF a hook is advisory rather than blocking, THEN the report SHALL clearly
   label it as advisory.
5. GIVEN hook adoption is staged, WHEN the roadmap is implemented, THEN early
   hooks SHALL be deterministic and advisory before later lifecycle gates become
   blocking.
6. GIVEN an agent-oriented hook runs, WHEN it evaluates agent work, THEN it
   SHALL check declared task IDs, evidence, changed files, and validation
   claims without treating unsupported agent prose as evidence.
7. GIVEN repeated waivers or noisy findings, WHEN hook metrics are reviewed,
   THEN the workflow SHALL support demoting, removing, or refining the hook.

### Requirement 9: Keep Security And Trust Boundaries Explicit

**User Story:** As a security-conscious operator, I want MCP prompts, tools,
resources, and hooks to avoid hidden instruction injection, so that spec
automation remains auditable.

#### Acceptance Criteria

1. GIVEN an MCP prompt, WHEN it embeds resources, THEN the generated messages
   SHALL identify which resources are included.
2. GIVEN tool-generated text, WHEN it is passed to an agent, THEN the tool
   SHALL distinguish observed facts, inferred diagnoses, and recommended
   actions.
3. IF a prompt or review packet includes untrusted document content, THEN it
   SHALL frame that content as data rather than instructions.
4. WHERE hooks execute local commands, THE SYSTEM SHALL document command scope,
   permissions, and failure behavior.

### Requirement 10: Measure Workflow Value

**User Story:** As a process owner, I want lightweight metrics for MCP-assisted
spec work, so that the workflow can be simplified if it adds overhead without
benefit.

#### Acceptance Criteria

1. GIVEN an MCP-assisted spec workflow, WHEN the workflow completes, THEN cycle
   time, lint findings, review findings, accepted findings, and closure risk
   can be recorded.
2. GIVEN repeated use, WHEN metrics are reviewed, THEN the team can identify
   which capabilities are useful, noisy, or unused.
3. IF a prompt, hook, or linter creates repeated low-value friction, THEN the
   workflow SHALL support removing or demoting it.

## Correctness Properties

- A completed task is never reported as valid unless acceptance criteria and
  evidence are present or an explicit waiver records residual risk.
- A closed spec is never reported as closure-ready while accepted durable
  behavior exists only in the temporary spec package.
- Old-format specs are detected but not migrated without a visible migration
  decision.
- MCP prompts never become the only path to the workflow; Skills plus tools and
  resources remain sufficient.
- Cheap-agent findings are advisory until reconciled by the lead agent or
  operator.

## Success Criteria

- A detailed MCP capability design exists.
- The spec clearly defines when MCP prompts are useful and when Skills are
  better.
- The spec defines linter, reconciliation, next-task, review-packet, and hook
  behavior.
- Implementation tasks are staged so an MVP can ship deterministic resources
  and linters before semantic agent offload.
- Security, governance, and client-support limits are explicit.

## Open Questions

- Should the MCP server live in this repository or be packaged as a separate
  reusable plugin?
- Should hooks be implemented as Git hooks, Codex hooks, an MCP tool callable
  from clients, or all three?
- Which clients in this environment expose MCP prompts well enough to justify
  prompt-first workflow entry points?
- Should review-packet execution be managed by this MCP server or only
  generated for a lead agent to dispatch?
