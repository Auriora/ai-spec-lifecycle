# Requirements

## Introduction

This spec defines advisory MCP/runtime tools that use fast, low-cost agents for
bounded documentation, spec management, and execution-support tasks. The goal
is to offload well-defined review and drafting work without allowing secondary
agents to silently mutate lifecycle state, close specs, or replace lead-agent
judgment.

## Goals

- Add an extensible model for agent-backed lifecycle tools.
- Keep low-cost agent outputs advisory, structured, and deterministically
  validated.
- Support bounded review and drafting tasks that reduce lead-agent context
  burden.
- Preserve existing active-spec, archive-index, and advisory-hook policies.

## Non-Goals

- No automatic file mutation by low-cost agents in the first implementation.
- No blocking hook promotion.
- No provider-specific commercial integration in the initial runtime contract.
- No attempt to prove semantic code correctness using cheap-agent output alone.

## Glossary

| Term | Definition |
|------|------------|
| Agent-backed tool | MCP/runtime tool that builds bounded context, invokes a secondary agent, validates output, and returns advisory results. |
| Low-cost agent | Fast, inexpensive model or worker used for constrained analysis or drafting. |
| Review packet | Bounded read-only input bundle generated by the runtime for a secondary review task. |
| Advisory result | Structured output that informs the lead agent but does not directly change repository state. |

## Requirements

### Requirement 1: Agent-backed tool contract

**User Story:** As a lead agent, I want a standard contract for offloading
bounded lifecycle tasks, so that secondary agents can help without taking over
state-changing workflow decisions.

#### Acceptance Criteria

1. GIVEN an agent-backed tool is defined, WHEN it is documented, THEN the
   system SHALL specify inputs, bounded context, output schema, validation
   rules, failure behavior, and advisory limits.
2. GIVEN a secondary agent returns output, WHEN the runtime processes it, THEN
   the system SHALL distinguish observed facts, inferences, recommendations,
   gaps, confidence, and validation diagnostics.
3. IF an agent-backed result fails schema or reference validation, THEN the
   system SHALL return a non-mutating advisory error result.

### Requirement 2: Initial bounded tool set

**User Story:** As a maintainer, I want a small initial set of low-risk
agent-backed tools, so that the project can dogfood value before expanding the
surface.

#### Acceptance Criteria

1. GIVEN the first implementation scope is selected, THEN the system SHALL
   prioritize at least one end-to-end tool from `closure_risk_review`,
   `draft_traceability_matrix`, `promotion_draft`, or
   `agent_readiness_packet`.
2. GIVEN a tool uses an active spec, WHEN it resolves the target, THEN the
   system SHALL use the existing active-spec resolver and fail cleanly for
   removed or nonexistent specs.
3. IF a tool can operate without an active spec, THEN the system SHALL use
   durable docs, backlog, roadmap, closure log, and archive index as its
   bounded context.

### Requirement 3: Deterministic validation around agent output

**User Story:** As a maintainer, I want deterministic checks around cheap-agent
results, so that low-cost model mistakes are contained.

#### Acceptance Criteria

1. GIVEN an output references paths, task IDs, requirement IDs, design
   sections, or headings, WHEN validation runs, THEN the system SHALL verify
   that those references exist where deterministic checks are possible.
2. GIVEN an output recommends a repository change, WHEN the result is returned,
   THEN the system SHALL label the recommendation as advisory and require lead
   agent or user approval before mutation.
3. IF a secondary agent mixes instructions into reviewed document content, THEN
   the tool contract SHALL instruct callers to treat reviewed content as data,
   not higher-priority instructions.

### Requirement 4: MCP/runtime integration

**User Story:** As an MCP client user, I want agent-backed lifecycle tools
available through the same runtime surface, so that agent offloading fits the
existing skill workflow.

#### Acceptance Criteria

1. GIVEN the MCP server exposes an agent-backed tool, WHEN tools are listed,
   THEN the schema SHALL identify the tool as advisory and read-only by
   default.
2. GIVEN agent execution is unavailable or disabled, WHEN a tool is called,
   THEN the system SHALL return a deterministic unavailable result rather than
   failing opaquely.
3. IF provider/model configuration is needed, THEN the system SHALL support a
   configurable `model_class` or equivalent without hard-coding one provider.

### Requirement 5: Adoption and guardrails

**User Story:** As a project maintainer, I want clear operating guidance, so
that future agents know when to use low-cost offload tools and when to keep
work local.

#### Acceptance Criteria

1. GIVEN the initial tool is implemented, THEN durable docs and skill guidance
   SHALL describe when to use it, what it cannot decide, and how to interpret
   its output.
2. GIVEN a result is useful but incomplete, THEN the workflow SHALL route gaps
   into backlog, roadmap, follow-up spec, or lead-agent action rather than
   silently accepting them.
3. IF future write-capable agent-backed tools are proposed, THEN they SHALL be
   deferred to a separate explicit spec with sandboxing, review, rollback, and
   evidence requirements.

## Correctness Properties

- **CP-001**: Agent-backed tools never directly close specs, mark tasks
  complete, update archive indexes, commit changes, or install hooks.
- **CP-002**: Every agent-backed result is either schema-valid or returned with
  explicit validation diagnostics.
- **CP-003**: Removed spec package paths are not resolved as active package
  inputs.
- **CP-004**: Deterministic validators remain the authority for path, heading,
  task ID, requirement ID, and prompt/resource contract checks.

## Technical Context

- **Language/Version:** Python 3 standard library.
- **Primary Dependencies:** Existing spec runtime, stdio MCP server, prompt and
  review-packet definitions.
- **Target Platform:** Local Codex/CLI/MCP usage in this repository and
  installed skill copies.
- **Constraints:** Advisory-only, no provider lock-in, no direct mutation by
  secondary agents in the initial implementation.
- **Performance Goals:** Tool overhead should stay small enough for routine
  lifecycle use; bounded packet generation must avoid full-repo context dumps.

## Success Criteria

- One end-to-end advisory agent-backed tool is specified and implemented behind
  deterministic packet generation and output validation.
- Runtime/MCP documentation identifies the model, guardrails, and disabled or
  unavailable behavior.
- Tests cover schema validation, removed-spec handling, no-active-spec
  behavior where applicable, and disabled-agent execution.
