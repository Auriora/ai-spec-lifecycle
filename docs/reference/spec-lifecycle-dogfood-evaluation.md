# Spec Lifecycle Dogfood Evaluation

This note records external verification and dogfood feedback for the
`spec-lifecycle-manager` skill, MCP server, runtime tools, and advisory hooks.
Use it to inform later hook-promotion and workflow-simplification decisions.

## Current Hook Policy

Lifecycle hooks remain advisory-only. External verification found the current
hooks useful and low-noise, but the project owner decided not to promote them
to blocking behavior. Any future blocking hook proposal must be handled as a
new focused spec with explicit approval, false-positive handling, and rollback
guidance.

## 2026-06-06 External Verification 1

Scope:

- Installed user-level skill sync.
- Host-level spec lifecycle MCP configuration.
- MCP tool visibility and representative tool calls.
- Advisory Codex hook configuration.
- Direct runtime CLI checks and unit tests.
- Alignment between durable docs, installed skill, MCP behavior, and hooks.

Result:

- **Blocking findings:** none.
- **Installed skill:** synced with repository source, ignoring local
  `__pycache__` artifacts.
- **MCP:** visible as `mcp__spec_lifecycle_manager`; `scan_specs`,
  `prompts_validate`, `archive_index`, `closure_check`, `task_context`, and
  `traceability_lookup` succeeded.
- **Hooks:** configured as advisory-only `PostToolUse` checks invoking the
  installed `codex_spec_lifecycle_hook.py`; no blocking behavior found.
- **Duplicate MCP:** no duplicate spec lifecycle MCP server found through the
  Agent Workbench plugin or cache.
- **CLI validation:** `scan`, `archive-index`, `prompts`, spec 012
  `closure-check`, and the full unit test suite passed.

Observed value:

- The skill worked best as a discipline mechanism that prevents agents from
  treating `tasks.md` as the source of truth.
- `task_context` and `traceability_lookup` were the most valuable MCP/runtime
  surfaces for avoiding implementation from task titles alone.
- The advisory hook caught useful package-shape issues without blocking work.
- Runtime checks were low-noise for inventory, closure readiness, prompt
  validation, and archive-index health.

Limits:

- Spec quality still matters; incomplete or older packages can produce noisy
  findings.
- Task metadata can become a checkbox exercise when evidence is not tied to
  actual implementation, tests, durable docs, or manual verification.
- The skill and runtime do not replace engineering judgment about whether
  implemented behavior is semantically correct.

Follow-up implications:

- Keep lifecycle hooks advisory until repeated dogfood evidence shows low
  false-positive risk.
- Consider selective future blocking only for narrow checks such as completed
  active-spec tasks without evidence or active specs missing core artifacts.
- Prioritize making task-context lookup harder to skip before implementation,
  especially for agents starting a task from `tasks.md`.
- Root `AGENTS.md` now exists for this repository; agents should review it
  before lifecycle work and ask for `/init` only once per session when
  repository instructions are absent.

## 2026-06-06 External Verification 2

Scope:

- Installed skill content sync.
- Host-level MCP configuration and tool visibility.
- Advisory hook behavior and duplicate MCP checks.
- Direct CLI validation.
- Review of durable wording against current governance state.

Result:

- **Blocking findings:** none.
- **Installed skill:** source and user-level installation were synced at
  content level, with only installed generated bytecode observed.
- **MCP:** visible with `scan_specs`, `prompts_validate`, `archive_index`,
  `closure_check`, `task_context`, and `traceability_lookup`; representative
  calls matched CLI behavior.
- **Hooks:** configured as advisory-only; wrapper uses the advisory severity
  profile, emits context only on diagnostics, and exits without blocking.
- **Duplicate MCP:** no duplicate spec lifecycle MCP registration found through
  Agent Workbench plugin configuration or cache.
- **CLI validation:** `scan`, `archive-index`, `prompts`, spec 012
  `closure-check`, and the full unit test suite passed.

Observed value:

- `next_task` and `task_context` were useful for resuming after reloads and
  avoiding tasks whose dependencies were not complete.
- Spec lint and closure checks caught structure and traceability issues without
  becoming noisy.
- Durable-doc promotion before closure improved implementation quality by
  making docs and closure part of delivery rather than cleanup.
- Archived-spec handling is clearer now that historical packages remain
  visible without polluting active lint.

Limits:

- The skill is still a governance and checklist helper; agents must reconcile
  code and tests manually.
- Deterministic closure readiness cannot catch every stale semantic statement
  in durable docs.
- The MCP and CLI surfaces overlap; `task_context`, `next_task`,
  `lint_spec_package`, and `closure_check` were the most useful day-to-day
  surfaces, while others are situational.

Follow-up implications:

- Keep hooks advisory-only for now.
- Remove generated bytecode from installed skill copies during the next
  explicit skill sync or install hygiene pass.
- Keep root `AGENTS.md` aligned with governance entry points and no-active-spec
  behavior as the skill evolves.
